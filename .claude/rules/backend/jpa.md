# JPA Rules

이 문서는 이 프로젝트에서 JPA 엔티티, 연관관계, 조회, 저장 로직을 다룰 때 따라야 할 규칙을 정의한다.

---

## 1. 기본 원칙

- JPA는 단순한 테이블 매핑 도구가 아니라 도메인 상태를 영속화하는 수단으로 사용한다.
- 엔티티는 단순 데이터 보관 객체가 아니라 상태와 행위를 함께 가지는 도메인 객체로 다룬다.
- 엔티티는 DTO와 역할이 다르므로, 엔티티에 DTO 변환 책임을 두지 않는다.
- 엔티티 변경은 곧 DB 스키마 및 기존 쿼리에 영향을 줄 수 있으므로 신중하게 다룬다.

---

## 2. 엔티티 설계 규칙

- 엔티티는 자신의 상태 변경 메서드를 가질 수 있다.
- 상태 변경, 도메인 규칙, 자기 일관성 검증은 엔티티 안에 둘 수 있다.
- 여러 엔티티를 조합하는 흐름, 외부 시스템 호출, Repository 조회가 필요한 로직은 서비스에서 처리한다.
- 엔티티에는 비즈니스적으로 의미 있는 메서드명을 사용한다.
- 의미 없는 setter 남발을 피한다.
- 무분별한 public 기본 생성자는 지양하고, JPA용 기본 생성자는 protected 수준을 우선 고려한다.
- 엔티티 생성 시점에 필요한 핵심 값은 생성자 또는 정적 팩토리 메서드로 받는 방식을 우선 고려한다.

예시:
- `changeStatus()`
- `approve()`
- `closeRecruitment()`
- `updateProfile()`

지양:
- `setStatus()`
- `setTitle()`
- `setValue()`

---

## 3. DTO와 엔티티 분리 규칙

- 엔티티를 API 요청/응답 객체처럼 사용하지 않는다.
- 컨트롤러에서 엔티티를 직접 반환하지 않는다.
- Response DTO의 `from(entity)` 패턴은 허용한다.
- 복잡한 생성/조합 로직은 서비스에서 처리한다.
- Request DTO의 `toEntity()`는 단순 생성인 경우에만 제한적으로 허용한다.
- 엔티티 안에 `toDto()`, `fromDto()` 같은 메서드를 두지 않는다.

---

## 4. 연관관계 설계 규칙

- 연관관계는 꼭 필요한 경우에만 추가한다.
- 객체 탐색이 실제로 필요한 경우에만 연관관계를 맺는다.
- 단순 조회 편의를 위해 무분별하게 양방향 연관관계를 만들지 않는다.
- 기본적으로 단방향 연관관계를 우선 고려한다.
- 양방향 연관관계가 꼭 필요할 때만 사용하고, 연관관계 주인을 명확히 관리한다.
- 컬렉션 연관관계는 정말 필요한 경우에만 둔다.
- 다대다(`@ManyToMany`)는 직접 사용하지 않고, 연결 엔티티를 분리하는 방식을 우선 사용한다.

---

## 5. Fetch 전략 규칙

- 연관관계 성능 문제를 항상 고려한다.
- ToOne 관계는 기본 EAGER 설정에 의존하지 말고, 명시적으로 LAZY 사용을 우선 고려한다.
- ToMany 관계는 기본적으로 LAZY를 유지한다.
- 필요한 데이터는 조회 시점에 fetch join, DTO 조회, 별도 조회 전략으로 가져온다.
- 단순히 편하다는 이유로 EAGER를 사용하지 않는다.
- 화면/API 응답에 필요한 데이터가 무엇인지 먼저 정하고 조회 방식을 선택한다.

원칙:
- 엔티티 설계와 조회 최적화는 분리해서 생각한다.
- "엔티티에 연관관계를 걸어두면 알아서 잘 가져오겠지"라고 가정하지 않는다.

---

## 6. 조회 규칙

- 목록 조회와 상세 조회를 같은 방식으로 처리하려고 하지 않는다.
- 목록 조회는 필요한 필드만 가져오는 DTO 조회를 우선 고려한다.
- 상세 조회는 필요한 연관 데이터 범위를 명확히 한 뒤 조회한다.
- 컨트롤러 바깥에서 지연 로딩이 우연히 터지도록 방치하지 않는다.
- 조회 성능이 중요한 경우 엔티티 전체 조회보다 DTO 프로젝션을 우선 검토한다.
- N+1 가능성이 있으면 fetch join 또는 조회 구조 변경을 먼저 검토한다.

---

## 7. 저장/수정 규칙

- 상태 변경은 가능한 한 엔티티 메서드를 통해 수행한다.
- 서비스는 유스케이스 흐름과 트랜잭션을 관리하고, 개별 상태 변경은 엔티티에 위임한다.
- 영속 상태 엔티티를 수정하는 경우, 단순 setter 나열보다 의도가 드러나는 메서드를 사용한다.
- 저장 로직은 "무조건 save 호출" 방식으로 생각하지 않는다.
- 이미 조회된 영속 엔티티의 변경은 변경 감지를 우선 활용한다.
- 생성과 수정 흐름을 구분해서 생각한다.

---

## 8. Cascade 및 orphanRemoval 규칙

- Cascade는 편의 기능이 아니라 생명주기 전파라는 점을 인식하고 사용한다.
- Aggregate 경계가 명확하고 함께 저장/삭제되어야 하는 경우에만 Cascade를 사용한다.
- 무분별한 `CascadeType.ALL` 사용을 지양한다.
- `orphanRemoval = true`는 부모-자식 생명주기가 강하게 묶여 있을 때만 사용한다.
- 연관 엔티티의 삭제가 실제로 의도한 동작인지 확인 없이 cascade 설정을 추가하지 않는다.

---

## 9. Soft Delete 규칙

- 이 프로젝트는 soft delete를 사용한다.
- `deletedAt = null`은 활성 데이터를 의미한다.
- 조회 시 soft delete 데이터가 포함되면 안 되는지 항상 먼저 판단한다.
- soft delete 엔티티를 조회 조건에서 빠뜨리지 않도록 주의한다.
- 실제 삭제와 soft delete를 혼동하지 않는다.

---

## 10. 트랜잭션 규칙

- 트랜잭션 경계는 서비스 계층에서 관리한다.
- 컨트롤러에서 트랜잭션 로직을 다루지 않는다.
- 하나의 유스케이스를 처리하는 동안 필요한 엔티티 조회, 상태 변경, 저장 흐름을 서비스 안에서 일관되게 관리한다.
- 읽기 전용 조회는 가능하면 readOnly 트랜잭션을 고려한다.
- 트랜잭션 범위를 불필요하게 넓히지 않는다.

---

## 11. 예외 및 검증 규칙

- 도메인 상태와 관련된 검증은 엔티티 또는 서비스에서 수행한다.
- 자기 상태만으로 판단 가능한 검증은 엔티티에서 처리할 수 있다.
- 다른 엔티티 조회나 외부 의존이 필요한 검증은 서비스에서 처리한다.
- 예외는 프로젝트 공통 예외 체계인 `CustomException(ErrorCode)`를 사용한다.

---

## 12. 금지 사항

- 엔티티를 API 응답으로 직접 반환하지 않는다.
- 엔티티에 DTO 변환 메서드를 만들지 않는다.
- `@ManyToMany`를 직접 사용하지 않는다.
- 무분별한 양방향 연관관계를 만들지 않는다.
- EAGER 기본값에 의존하지 않는다.
- 의미 없는 setter 중심으로 엔티티를 수정하지 않는다.
- 엔티티 변경 후 스키마 영향 검토 없이 넘어가지 않는다.
- soft delete 엔티티를 일반 조회에서 무심코 노출하지 않는다.

---

## 13. 작업 시 체크리스트

JPA 관련 작업을 할 때 아래를 먼저 확인한다.

### 엔티티 추가/수정 시
- 이 필드가 정말 엔티티에 필요한가?
- 값 객체로 분리할 여지는 없는가?
- 상태 변경 메서드가 필요한가?
- setter 대신 의도 있는 메서드로 표현할 수 있는가?
- 스키마 변경이 필요한가?
- 기존 쿼리나 연관관계에 영향이 있는가?

### 연관관계 추가 시
- 연관관계가 정말 필요한가?
- 단방향으로 충분한가?
- 양방향이 꼭 필요한가?
- fetch 전략은 안전한가?
- N+1 위험은 없는가?

### 조회 로직 작성 시
- 엔티티 조회가 필요한가, DTO 조회가 더 적절한가?
- 목록/상세 조회 목적에 맞는가?
- 필요한 연관 데이터만 가져오고 있는가?
- soft delete 데이터가 섞이지 않는가?

### 저장/수정 로직 작성 시
- 서비스가 유스케이스 흐름을 담당하고 있는가?
- 상태 변경이 엔티티 메서드로 표현되는가?
- 변경 감지를 활용할 수 있는가?
- cascade 설정이 정말 필요한가?

---

## 14. 이 프로젝트에서의 우선순위

이 프로젝트에서는 아래 우선순위를 따른다.

1. 도메인 의미가 드러나는 엔티티 설계
2. DTO와 엔티티 역할 분리
3. 서비스에서 유스케이스 흐름 관리
4. 연관관계 남발 방지
5. 조회 성능과 N+1 방지
6. 스키마 영향에 대한 신중한 검토
